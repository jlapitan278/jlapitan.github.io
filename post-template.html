Detection Engineering

Writing Detection Rules That Actually Fire in Production

Josh · March 22, 2025 · 8 min read

There's a familiar pattern in detection engineering: you write a rule in your lab, it catches the simulated attack perfectly, you ship it to prod — and then it either never fires or fires every thirty seconds on benign noise. Both outcomes are useless. Here's the mental model I've built for writing rules that land somewhere useful in the middle.

Start with the data, not the technique

The first mistake most rule writers make is leading with a MITRE technique and then trying to find data to match. Flip that. Start by understanding the shape of your logs — what fields exist, what values are normal, and where the noise lives — before you write a single line of SPL or KQL.

For any new log source, I spend time writing exploratory queries with no detection intent. Just answering questions: what does a normal process_create event look like? What's the parent_process_name distribution for powershell.exe? This baseline work is what makes a tight rule possible.

The anatomy of a tunable rule

A production-worthy rule has three zones you can adjust independently:

  • The anchor — the high-signal condition that is nearly always true for the malicious behavior. Keep this narrow.
  • The allowlist — an explicit set of known-good conditions to subtract. This is where false-positive tuning lives.
  • The threshold or aggregation — how many events, from how many hosts, in what time window, before the alert fires.

Separating these concerns means you can tune the allowlist without touching the core logic, and adjust sensitivity without rewriting the detection.

An example: detecting suspicious PowerShell downloads

Here's a minimal Splunk rule for T1059.001 (PowerShell) combined with T1105 (Ingress Tool Transfer): 

index=endpoint sourcetype=WinEventLog:4688
  CommandLine="*Invoke-WebRequest*" OR CommandLine="*iwr *" OR CommandLine="*curl *"
  ParentProcessName!="C:\\Windows\\System32\\msiexec.exe"
  NOT [inputlookup ps_download_allowlist.csv | fields CommandLine]
| stats count by ComputerName, User, CommandLine
| where count >= 1

The allowlist lookup (ps_download_allowlist.csv) is where you capture known software installers that use PowerShell downloads legitimately. Adding to the allowlist is a one-line CSV edit, not a rule change.

Shipping vs. tuning mode

A rule in "tuning mode" runs silently — generating notable events but not pages. It earns its way to production by going a week without a false positive you can't explain.

I tag every new rule with a tuning status field and review the noisy ones every Monday. A rule graduates to "production" only when the signal-to-noise ratio is acceptable and the allowlist is stable. Most rules spend two to four weeks in tuning.

Final thought

Detection is a product, not a configuration file. It needs the same iteration loop as software: write, ship to a staging environment, measure outcomes, refine. The engineers who write the best detections aren't the ones who know the most TTPs — they're the ones who are most comfortable working with dirty, real-world log data.
